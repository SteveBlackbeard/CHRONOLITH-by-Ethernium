# Change Contract: Release 3.2.1

## Summary

Raise the version from `3.2.0` to `3.2.1` across `VERSION` and the four
`pyproject.toml` files, and refresh the golden baseline to match.

## Scope

Files expected to change:

- `VERSION`
- `pyproject.toml`
- `chronolith-lite/pyproject.toml`
- `chronolith-pro/pyproject.toml`
- `chronolith-omega/pyproject.toml`
- `CHANGELOG.md`
- `.chronolith/golden-baseline.json`

Out of scope:

- runtime behaviour. This release contains fixes already reviewed and merged;
  the bump itself changes nothing but the number.

## Reason

A patch release for defects found by running the software rather than reading
it. The headline is that `chain` had never once worked and the Sentinel job had
never once succeeded — both shipped, both documented, neither executed by any
test until now.

The baseline governs `pyproject.toml`, so a version bump is drift by design and
requires this contract. That is the guard behaving correctly, not an obstacle.

## Risk

Risk level: `low`

Potential failure modes:

- **package metadata drift**: the four editions must carry the same version.
  Verified by reading each `pyproject.toml` after the bump.
- **state/hash mismatch**: the baseline is refreshed from LF content and
  checked against `git cat-file`, not against the local working tree, because
  this machine writes CRLF and would otherwise encode hashes CI cannot match.

## Verification

```text
python scripts/audit_all.py
python scripts/golden_baseline.py verify
python scripts/health_guard.py --strict
```

Publication is verified after the fact: a release published through Trusted
Publishing carries a PyPI attestation naming this repository and workflow.
Absence of that attestation means the release fell back to a token and the
publisher configuration is incomplete.

## Rollback

```text
git revert <commit>
```

No published artefact needs withdrawing: PyPI versions are immutable, so a bad
3.2.1 is superseded by 3.2.2 rather than replaced.
