# CONTINUITY LEGACY: Governance & Maintainers 🏛️



To solve the "Single Maintainer" risk, this project follows an open, merit-based governance model. Documentation and continuity are as important as the code.



## 👥 Meet the Maintainers

- **Founding Maintainer**: @Steveblackbeard (Ethernium)

- **Status**: Active



## 🎖️ Maintainer Roles

1. **Guardian**: Responsible for the continuity core and validation scripts.

2. **Archivist**: Ensures all `DECISIONS_LOG.md` and `TIMELINE.md` entries are canonical and high-quality.

3. **Ambassador**: Manages international translations and community support.



## 🚀 How to become a Maintainer

We follow the **Continuity Meritocracy**:

1. **Contributor**: Submit 3+ Pull Requests that strictly pass the `--strict` cycle.

2. **Validator**: Review 5+ PRs ensuring document parity is maintained.

3. **Maintainer**: Nominated by an existing maintainer after demonstrating deep alignment with the project's "Continuity-First" philosophy.



## 💀 Bus Factor & Succession Plan

If the primary maintainer is inactive for more than 90 days:

1. The project's own `LIVE_HANDOFF.md` serves as the official state transfer document.

2. The most active **Validator** or **Contributor** is encouraged to fork or request admin access to continue the legacy.

3. The codebase is designed to be **Self-Governing**; the scripts themselves guide the next operator.
