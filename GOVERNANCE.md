# CONTINUITY LEGACY: Project Governance & Resilience 

This document defines how **CONTINUITY LEGACY** is maintained and how its longevity is ensured beyond a single maintainer.

## 1. The Continuity Principle (Project Longevity)
The core risk of a "1-maintainer" project is the loss of strategic context. CONTINUITY LEGACY mitigates this by applying its own framework to its own development:
- **Canonical Logs**: Every architectural decision is recorded in .continuity/DECISIONS_LOG.md.
- **State Manifests**: Every project phase is codified in STATE.json.
- **Succession**: In the event of maintainer inactivity, any contributor can use the existing continuity files as a "Lifeboat" to resume development with 100% contextual fidelity.

## 2. Decision Making
- **Strategic Direction**: Managed by the Project Lead (SteveBlackbeard).
- **Technical Proposals**: Evaluated via the **Decision Engine** (Pro version) to ensure alignment with the "Universal Core" principles.
- **Strict Mode**: No contribution will be accepted unless it passes the --strict validation cycle.

## 3. Contributing
We welcome developers who value technical excellence:
- **Lite**: Focus on performance and zero-dependency core logic.
- **Pro**: Focus on tactical automation and user-driven features.
- **Omega**: Focus on AI/ML integration and cognitive mapping.
